                                <h1 id="event-system">Event system</h1>

This package is heavily based around events, which makes it incredibly flexible.



By default, the events are configured in such a way that the package works like this:



<ul>
<li>A request comes in for a tenant route and hits an identification middleware</li>
<li>The identification middleware finds the correct tenant and runs</li>
</ul>

```
<code class="language-php">$this-&gt;tenancy-&gt;initialize($tenant);
</code>
```


<ul>
<li>The `Stancl\Tenancy\Tenancy` class sets the `$tenant` as the current tenant and fires a `TenancyInitialized` event</li>
<li>The `BootstrapTenancy` class catches the event and executes classes known as <a href="https://tenancyforlaravel.com/docs/v3/tenancy-bootstrappers">tenancy bootstrappers</a>.</li>
<li>The tenancy bootstrappers make changes to the application to make it "scoped" to the current tenant. This by default includes:

<ul>
<li>Switching the database connection</li>
<li>Replacing `CacheManager` with a scoped cache manager</li>
<li>Suffixing filesystem paths</li>
<li>Making queues store the tenant id &amp; initialize tenancy when being processed</li>
</ul></li>
</ul>

Again, all of the above is configurable. You might even disable all tenancy bootstrappers, and just use tenant identification and scope your app manually around the tenant stored in `Stancl\Tenancy\Tenancy`. The choice is yours.



<h1 id="tenancyserviceprovider">TenancyServiceProvider</h1>

This package comes with a very convenient service provider that's added to your application when you install the package. This service provider is used for mapping listeners to events specific to the package and is the place where you should put any tenancy-specific service container calls — to not pollute your AppServiceProvider.



Note that you can register listeners to this package's events <strong>anywhere you want</strong>. The event/listener mapping in the service provider exists only to make your life easier. If you want to register the listeners manually, like in the example below, you can.



```
<code class="language-php">Event::listen(TenancyInitialized::class, BootstrapTenancy::class);
</code>
```


<h1 id="bootstrapping-tenancy">Bootstrapping tenancy</h1>

By default, the `BootstrapTenancy` class is listening to the `TenancyInitialized` event (exactly as you can see in the example above). That listener will execute the configured tenancy bootstrappers to transition the application into the tenant's context. You can read more about this on the <a href="https://tenancyforlaravel.com/docs/v3/tenancy-bootstrappers">tenancy bootstrappers</a> page.



Conversely, when the `TenancyEnded` event fires, the `RevertToCentralContext` event transitions the app back into the central context.



<h1 id="job-pipelines">Job pipelines</h1>

You may want to use job pipelines even in projects that don't use this package — I think they're a cool concept so they're extracted into a separate package: <a href="https://github.com/stancl/jobpipeline">github.com/stancl/jobpipeline</a>



The `JobPipeline` is a simple, yet <strong>extremely powerful</strong> class that lets you <strong>convert any (series of) jobs into event listeners.</strong>



You may use a job pipeline like any other listener, so you can register it in the `TenancyServiceProvider`, `EventServiceProvider` using the `$listen` array, or in any other place using `Event::listen()` — up to you.



<h2 id="creating-job-pipelines">Creating job pipelines</h2>

To create a job pipeline, start by specifying the jobs you want to use:



```
<code class="language-php">use Stancl\JobPipeline\JobPipeline;
use Stancl\Tenancy\Jobs\{CreateDatabase, MigrateDatabase, SeedDatabase};

JobPipeline::make([
    CreateDatabase::class,
    MigrateDatabase::class,
    SeedDatabase::class,
])
</code>
```


Then, specify what variable you want to pass to the jobs. This will usually come from the event.



```
<code class="language-php">use Stancl\JobPipeline\JobPipeline;
use Stancl\Tenancy\Jobs\{CreateDatabase, MigrateDatabase, SeedDatabase};
use Stancl\Tenancy\Events\TenantCreated;

JobPipeline::make([
    CreateDatabase::class,
    MigrateDatabase::class,
    SeedDatabase::class,
])-&gt;send(function (TenantCreated $event) {
    return $event-&gt;tenant;
})
</code>
```


Next, decide if you want to queue the pipeline. By default, pipelines are synchronous (= not queued).



<p>If you <strong>do</strong> want pipelines to be queued by default, you can do that by setting a static property:
`\Stancl\JobPipeline\JobPipeline::$shouldBeQueuedByDefault = true;`</p>

```
<code class="language-php">use Stancl\Tenancy\Events\TenantCreated;
use Stancl\JobPipeline\JobPipeline;
use Stancl\Tenancy\Jobs\{CreateDatabase, MigrateDatabase, SeedDatabase};

JobPipeline::make([
    CreateDatabase::class,
    MigrateDatabase::class,
    SeedDatabase::class,
])-&gt;send(function (TenantCreated $event) {
    return $event-&gt;tenant;
})-&gt;shouldBeQueued(true),
</code>
```


Finally, convert the pipeline to a listener and bind it to an event:



```
<code class="language-php">use Stancl\Tenancy\Events\TenantCreated;
use Stancl\JobPipeline\JobPipeline;
use Stancl\Tenancy\Jobs\{CreateDatabase, MigrateDatabase, SeedDatabase};
use Illuminate\Support\Facades\Event;

Event::listen(TenantCreated::class, JobPipeline::make([
    CreateDatabase::class,
    MigrateDatabase::class,
    SeedDatabase::class,
])-&gt;send(function (TenantCreated $event) {
    return $event-&gt;tenant;
})-&gt;shouldBeQueued(true)-&gt;toListener());
</code>
```


Note that you can use job pipelines even for converting single jobs to event listeners. That's useful if you have some logic in job classes and don't want to create listener classes just to be able to run these jobs as a result of an event being fired.



<h1 id="available-events">Available events</h1>

Note: Some database events (`DatabaseMigrated`, `DatabaseSeeded`, `DatabaseRolledback` and possibly others) are <strong>fired in the tenant context.</strong> Depending on how your application bootstraps tenancy, you might need to be specific about interacting with the central database in these events' listeners — that is, if you need to.



Note: All events are located in the `Stancl\Tenancy\Events` namespace.



<h3 id="tenancy"><strong>Tenancy</strong></h3>

<ul>
<li>`InitializingTenancy`</li>
<li>`TenancyInitialized`</li>
<li>`EndingTenancy`</li>
<li>`TenancyEnded`</li>
<li>`BootstrappingTenancy`</li>
<li>`TenancyBootstrapped`</li>
<li>`RevertingToCentralContext`</li>
<li>`RevertedToCentralContext`</li>
</ul>

Note the difference between <em>initializing tenancy and bootstrapping</em> tenancy. Tenancy is initialized when a tenant is loaded into the `Tenancy` object. Whereas bootstrapping happens <strong>as a result of initialization</strong> — if you're using automatic tenancy, the `BootstrapTenancy` class is listening to the `TenancyInitialized` event and after it's done executing bootstrappers, it fires an event saying that tenancy was bootstrapped. You want to use the bootstrapped event if you want to execute something <strong>after the app has been transitioned to the tenant context.</strong>
