                                <h1 id="single-database-tenancy">Single-database tenancy</h1>

Single-database tenancy comes with lower devops complexity, but larger code complexity than multi-database tenancy, since you have to scope things manually, and won't be able to integrate some third-party packages.



It is preferable when you have too many shared resources between tenants, and don't want to make too many cross-database queries.



To use single-database tenancy, make sure you disable the `DatabaseTenancyBootstrapper` which is responsible for switching database <strong>connections</strong> for tenants.



You can still use the other <a href="https://tenancyforlaravel.com/docs/v3/tenancy-bootstrappers">tenancy bootstrappers</a> to separate tenant caches, filesystems, etc.



Also make sure you have disabled the database creation jobs (`CreateDatabase`, `MigrateDatabase`, `SeedDatabase` ...) from listening to the `TenantCreated` event.



<h1 id="concepts">Concepts</h1>

In single-database tenancy, there are 4 types of models:



<ul>
<li>your <strong>Tenant</strong> model</li>
<li>primary models — models that <strong>directly</strong> belongTo tenants</li>
<li>secondary models — models that <strong>indirectly</strong> belongTo tenants

<ul>
<li>e.g. <strong>Comment</strong> belongsTo <strong>Post</strong> belongsTo <strong>Tenant</strong></li>
<li>or more complex, <strong>Vote</strong> belongsTo <strong>Comment</strong> belongsTo <strong>Post</strong> belongsTo <strong>Tenant</strong></li>
</ul></li>
<li>global models — models that are <strong>not scoped</strong> to any tenant whatsoever</li>
</ul>

To scope your queries correctly, apply the `Stancl\Tenancy\Database\Concerns\BelongsToTenant` trait <strong>on primary models</strong>. This will ensure that all calls to your parent models are scoped to the current tenant, and that <strong>calls to their child relations are scoped through the parent relationships</strong>.



And that's it. Your models are now automatically scoped to the current tenant, and not scoped at all when there's no current tenant (e.g. in a central admin panel).



However, there's one edge case to keep in mind. Consider the following set-up:



```
<code class="language-php">class Post extends Model
{
    use BelongsToTenant;

    public function comments()
    {
        return $this-&gt;hasMany(Comment::class);
    }
}

class Comment extends Model
{
    public function post()
    {
        return $this-&gt;belongsTo(Post::class);
    }
}
</code>
```


Looks correct, but you might still accidentally access another tenant's comments.



If you use this:



```
<code class="language-php">Comment::all();
</code>
```


then the model has no way of knowing how to scope that query, since it doesn't directly belong to the tenant. Also note that in practice, you really shouldn't be doing this much. You should ideally access secondary models through parent models in every single case.



However, sometimes you might have a use case where you <strong>really need to do that</strong> in the tenant context. For that reason, we also provide you with a `BelongsToPrimaryModel` trait, which lets you scope calls like the one above to the current tenant, by loading the parent relationship — which gets automatically scoped to the current tenant — on them.



So, to give you an example, you would do this:



```
<code class="language-php">class Comment extends Model
{
    use BelongsToPrimaryModel;

    public function getRelationshipToPrimaryModel(): string
    {
        return 'post';
    }

    public function post()
    {
        return $this-&gt;belongsTo(Post::class);
    }
}
</code>
```


And this will automatically scope the `Comment::all()` call to the current tenant. Note that the limitation of this is that you <strong>need to be able to define a relationship to a primary model</strong>, so if you need to do this on the "Vote" in <em><strong>Vote</strong> belongsTo <strong>Comment</strong> belongsTo <strong>Post</strong> belongsTo <strong>Tenant</strong>,</em> you need to define some strange relationship. Laravel supports `HasOneThrough`, but not `BelongsToThrough`, so you'd need to do some hacks around that. For that reason, I recommend avoiding these `Comment::all()`-type queries altogether.



<h1 id="database-considerations">Database considerations</h1>

<h3 id="unique-indexes">Unique indexes</h3>

If you'd have a unique index such as:



```
<code class="language-php">$table-&gt;unique('slug');
</code>
```


in a standard non-tenant, or multi-database-tenant, application, you need to scope this unique index to the tenant, meaning you'd do <strong>this on primary models:</strong>



```
<code class="language-php">$table-&gt;unique(['tenant_id', 'slug']);
</code>
```


and this on <strong>secondary models:</strong>



```
<code class="language-php">// Imagine we're in a 'comments' table migration

$table-&gt;unique(['post_id', 'user_id']);
</code>
```


<h3 id="validations">Validation</h3>

The `unique` and `exists` validation rules of course aren't scoped to the current tenant, so you need to scope them manually like this:



```
<code class="language-php">Rule::unique('posts', 'slug')-&gt;where('tenant_id', tenant('id'));
</code>
```


If that feels like a chore, you may use the `Stancl\Tenancy\Database\Concerns\HasScopedValidationRules` trait on your custom <a href="https://tenancyforlaravel.com/docs/v3/tenants">Tenant</a> model to add methods for these two rules.



You'll be able to use these two methods:



```
<code class="language-php">// You may retrieve the current tenant using the tenant() helper.
// $tenant = tenant();

$rules = [
    'id' =&gt; $tenant-&gt;exists('posts'),
    'slug' =&gt; $tenant-&gt;unique('posts'),
]
</code>
```


<h3 id="low-level-database-queries">Low-level database queries</h3>

And the final thing to keep in mind is that `DB` facade calls, or any other types of direct database queries, of course won't be scoped to the current tenant.



The package can only provide scoping logic for the abstraction logic that Eloquent is, it can't do anything with low level database queries.



Be careful with using them.



<h2 id="making-global-queries">Making global queries</h2>

To disable the tenant scope, simply add `withoutTenancy()` to your query.



<h2 id="customizing-the-column-name">Customizing the column name</h2>

If you'd like to customize the column name to use e.g. `team_id` instead of `tenant_id` — if that makes more sense given your business terminology — you can do that by setting this static property in a service provider or some such class:



```
<code class="language-php">use Stancl\Tenancy\Database\Concerns\BelongsToTenant;

BelongsToTenant::$tenantIdColumn = 'team_id';
</code>
```


Note that this is universal to all your primary models, so if you use `team_id` somewhere, you use it everywhere — you can't use both `team_id` and `tenant_id`.
