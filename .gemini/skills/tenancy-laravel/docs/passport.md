                                <h1 id="laravel-passport">Laravel Passport</h1>

<blockquote>
  <strong>Tip:</strong> If you just want to write an SPA application but don't need an API for some other use (e.g., a mobile app), you can avoid a lot of the complexity of writing SPAs by using <a href="https://inertiajs.com/">Inertia.js</a>.


  
  <strong>Another tip:</strong> Using Passport only in the central application doesn't require any additional configuration. You can just install it following <a href="https://laravel.com/docs/9.x/passport">the official Laravel Passport documentation</a>.


</blockquote>

<h2 id="using-passport-in-the-tenant-application-only"><strong>Using Passport in the tenant application only</strong></h2>

<blockquote>
  <strong>Note:</strong> Don't use the `passport:install` command. The command creates the encryption keys &amp; two clients in the central application. Instead of that, we'll generate the keys and create the clients manually later.


</blockquote>

To use Passport inside the tenant part of your application, you may do the following.



<ol>
<li>Publish the Passport migrations by running `php artisan vendor:publish --tag=passport-migrations` and move them to your tenant migration directory (`database/migrations/tenant/`).

</li>
<li>Publish the Passport config by running `php artisan vendor:publish --tag=passport-config`. If you're using Passport 10.x, make Passport use the default database connection by setting the storage database connection to `null`. The `passport:keys` command puts the keys in the `storage/` directory by default – you can change that by setting the key path in the config.



```
<code class="language-php">return [
    'storage' =&gt; [ // Needed only when using Passport 10.x
        'database' =&gt; [
            'connection' =&gt; null,
        ],
    ],
    'key_path' =&gt; env('OAUTH_KEY_PATH', 'storage') // This is optional
];
</code>
```
</li>
<li>Prevent Passport migrations from running in the central application by adding `Passport::ignoreMigrations()` to the `register()` method in your `AuthServiceProvider`.

</li>
<li>If you're using Passport 10.x, register the Passport routes in your `AuthServiceProvider` by adding the following code to the provider's `boot()` method:

</li>
</ol>

```
<code class="language-php">    Passport::routes(null, ['middleware' =&gt; [
        InitializeTenancyByDomain::class, // Or other identification middleware of your choice
        PreventAccessFromCentralDomains::class,
    ]]);
</code>
```


<ol start="5">
<li>If you're using Passport 11.x, disable the automatic Passport route registering in your `AuthServiceProvider` by adding `Passport::ignoreRoutes();` to the `register()` method. Then, register the Passport routes manually by adding the following code to the `boot()` method:</li>
</ol>

```
<code class="language-php">    Route::group([
        'as' =&gt; 'passport.',
        'middleware' =&gt; [
            InitializeTenancyByDomain::class, // Use tenancy initialization middleware of your choice
            PreventAccessFromCentralDomains::class,
        ],
        'prefix' =&gt; config('passport.path', 'oauth'),
        'namespace' =&gt; 'Laravel\Passport\Http\Controllers',
    ], function () {
        $this-&gt;loadRoutesFrom(__DIR__ . "/../../vendor/laravel/passport/src/../routes/web.php");
    });
</code>
```


<ol start="6">
<li>Apply Passport migrations by running `php artisan tenants:migrate`.

</li>
<li>Set up <a href="#passport-encryption-keys">the encryption keys</a>.

</li>
</ol>

<h2 id="using-passport-in-both-the-tenant-and-the-central-application"><strong>Using Passport in both the tenant and the central application</strong></h2>

To use Passport in both the tenant and the central application:



<ol>
<li>Follow <a href="#using-passport-in-the-tenant-application-only">the steps for using Passport in the tenant appliction</a>.

</li>
<li>Copy the Passport migrations to the central application, so that the Passport migrations are in both the central and the tenant application.

</li>
<li>Remove `Passport::ignoreMigrations()` from the `register()` method in your `AuthServiceProvider` (if it is there).

</li>
<li>In your `AuthServiceProvider`'s `boot()` method (where you registered the Passport routes), add the `'universal'` middleware to the Passport routes, and remove the `PreventAccessFromCentralDomains::class` middleware. The related code in your `boot()` method should look like this:

</li>
</ol>

```
<code class="language-php">// Passport 10.x
Passport::routes(null, ['middleware' =&gt; [
    'universal',
    InitializeTenancyByDomain::class
]]);

// Passport 11.x
Route::group([
    'as' =&gt; 'passport.',
    'middleware' =&gt; [
        'universal',
        InitializeTenancyByDomain::class
    ],
    'prefix' =&gt; config('passport.path', 'oauth'),
    'namespace' =&gt; 'Laravel\Passport\Http\Controllers',
], function () {
    $this-&gt;loadRoutesFrom(__DIR__ . "/../../vendor/laravel/passport/src/../routes/web.php");
});
</code>
```


<ol start="5">
<li>Enable <a href="https://tenancyforlaravel.com/docs/v3/features/universal-routes">universal routes</a> to make Passport routes accessible to both apps.</li>
</ol>

<h2 id="passport-encryption-keys"><strong>Passport encryption keys</strong></h2>

<h3 id="shared-keys"><strong>Shared keys</strong></h3>

To generate a single Passport key pair for the whole app, create Passport clients for your tenants by adding the following code to your <a href="https://tenancyforlaravel.com/docs/v3/configuration/#seeder-parameters">tenant database seeder</a>.



```
<code class="language-php">public function run()
{
    $client = new ClientRepository();

    $client-&gt;createPasswordGrantClient(null, 'Default password grant client', 'http://your.redirect.path');
    $client-&gt;createPersonalAccessClient(null, 'Default personal access client', 'http://your.redirect.path');
}
</code>
```


<em>You can set your tenant database seeder class in `config/tenancy.php` file at `seeder_parameters` key.</em>



Then, seed the database and generate the key pair by running `php artisan passport:keys`.



<h3 id="tenant-specific-keys"><strong>Tenant-specific keys</strong></h3>

<blockquote>
  <strong>Note:</strong> The security benefit of doing this is negligible since you're likely already using the same `APP_KEY` for all tenants. This is a relatively complex approach, so before implementing it, make sure you really want it. <strong>Using shared keys instead is strongly recommended.</strong>


  
  <strong>Warning:</strong> The usage of tenant specific keys has not been fully tested. <a href="https://github.com/stancl/tenancy-docs/edit/master/source/docs/v3/integrations/passport.blade.md">Feel free to contribute to this section.</a>


</blockquote>

If you want to use a unique Passport key pair for each tenant, there are multiple ways to store and load tenant Passport keys. The most straightforward way is to store them in the `Tenant model` and load them into the Passport configuration using the <a href="https://tenancyforlaravel.com/docs/v3/features/tenant-config">Tenant Config</a> feature. Then, you can access the keys like `$tenant-&gt;passport_public_key`.



To achieve that, enable the <a href="https://tenancyforlaravel.com/docs/v3/features/tenant-config">Tenant Config</a> feature, and configure the storage-to-config mapping in the `boot` method of your `TenancyServiceProvider` this way:



```
<code class="language-php">\Stancl\Tenancy\Features\TenantConfig::$storageToConfigMap = [
    'passport_public_key' =&gt; 'passport.public_key',
    'passport_private_key' =&gt; 'passport.private_key',
];
</code>
```
