                                <h1 id="compared-to-other-packages">Compared to other packages</h1>

<h2 id="hyn-multi-tenancy">hyn/multi-tenancy</h2>

This package intends to provide you with the necessary tooling for adding multi-tenancy <strong>manually</strong> to your application. It will give you model traits, classes for creating tenant databases, and some additional tooling.



It would have been a good option for when you want to implement multi-tenancy manually, but:



<ul>
<li>It isn't being actively developed — no features have been added in the past ~year</li>
<li>It makes testing a nightmare.



Over the last few months, I've received this feedback:



<blockquote>
  But, I still can't run any tests in Hyn, and had some queuing problems I'm still nervous about.


  
  At the moment, our app is using latest Laravel and latest hyn/tenancy. The only thing I don't like about it is that tests are extremely fragile to the point where I don't dare mess with anything for risk of breaking everything


  
  By the way, this package is awesome! It's so much better than the hyn alternative which is a bit of a mess in my opinion... It is a pity that I did not come across it in the first place.


</blockquote>

I'm not sharing this to intentionally make hyn/multi-tenancy bad, but <strong>be very careful if you decide to go with that package</strong>.

</li>
</ul>

<h2 id="tenancy-tenancy">tenancy/tenancy</h2>

This package intends to provide you with a framework for building your own multi-tenancy implementation. The documentation is quite lacking, so I couldn't get a too detailed idea of what it does, but from my understanding, it gives you things like events which you can use to build your own multi-tenancy logic.



If you want the absolute highest flexibility and would otherwise build your own implementation, looking into this package might be useful.



However, if you're looking for a package that will help you make a multi-tenant project quickly, this is probably not the right choice.



<h2 id="tenacy-laravel-multitenancy">spatie/laravel-multitenancy</h2>

This package is a very simple implementation of multi-tenancy.



It does the same thing as stancl/tenancy v2, but with far fewer features out of the box.



The only benefit I see in this package compared to v2 of stancl/tenancy is that it uses Eloquent out of the box, which makes things like Cashier integration easier. But, that's irrelevant since we're in v3 already and v3 uses Eloquent.



So, I suggest you consider this package only if you value simplicity for some reason, and aren't building anything with any amount of complexity and need for "business features".



<h2 id="stancl-tenancy">stancl/tenancy</h2>

In my — biased, of course, but likely true as well — opinion, this package is the absolute best choice for the vast majority of applications.



The only packages I'd realistically consider are mine (of course) and tenancy/tenancy if you need something <strong>very</strong> custom, though I don't see the reason for that in 99% of applications.



This package attempts to be about as flexible as tenancy/tenancy, but also provide you with the absolute largest amount of out-of-the-box features and other tooling. It continues its path as the first package to have been using the automatic approach with adding many more features — most of which are "enterprise" features, in v3.



To give you an incomplete-but-good-enough list of features, this package supports:



<ul>
<li>Multi-database tenancy

<ul>
<li>creating databases

<ul>
<li>MySQL</li>
<li>PostgreSQL</li>
<li>PostgreSQL (schema mode)</li>
<li>SQLite</li>
</ul></li>
<li>creating database users</li>
<li>automatically switching the database</li>
<li>CLI commands, with more features than e.g. spatie/laravel-multitenancy

<ul>
<li>migrate</li>
<li>migrate:fresh</li>
<li>seed</li>
</ul></li>
</ul></li>
<li>Single-database tenancy

<ul>
<li>model traits with global scopes</li>
</ul></li>
<li>Rich event system</li>
<li><strong>Very high testability</strong></li>
<li>Automatic tenancy

<ul>
<li>tenancy bootstrappers to switch:

<ul>
<li>database connections</li>
<li>redis connections</li>
<li>cache tags</li>
<li>filesystem roots</li>
<li>queue context</li>
</ul></li>
</ul></li>
<li>Manual tenancy

<ul>
<li>model traits</li>
</ul></li>
<li>Out of the box tenant identification

<ul>
<li>domain identification</li>
<li>subdomain identification</li>
<li>path identification</li>
<li>request data identification</li>
<li>middleware classes for the methods above</li>
<li>CLI argument identification</li>
<li>manual identification (e.g. in tinker)</li>
</ul></li>
<li>Integration with many packages

<ul>
<li>spatie/laravel-medialibrary</li>
<li>spatie/laravel-activitylog</li>
<li>Livewire</li>
<li>Laravel Nova

<ul>
<li>for managing tenants</li>
<li>for <strong>using inside the tenant application</strong></li>
</ul></li>
<li>Laravel Horizon</li>
<li>Laravel Telescope</li>
<li>Laravel Passport</li>
</ul></li>
<li><strong>Syncing users (or any other database resources) between multiple tenant databases</strong></li>
<li>Dependency injection of the current tenant</li>
<li>Tenant <strong>user impersonation</strong></li>
<li><strong>Cached tenant lookup</strong>, universal for all tenant resolvers</li>
</ul>
