                                <h1 id="configuration">Configuration</h1>

The package is highly configurable. This page covers what you can configure in the `config/tenancy.php` file, but note that many more things are configurable. Some things can be changed by extending classes (e.g. the `Tenant` model), and <strong>many</strong> things can be changed using static properties. These things will <em>usually</em> be mentioned on the respective pages of the documentation, but not every time. For this reason, don't be afraid to dive into the package's source code — whenever the class you're using has a `public static` property, <strong>it's intended to be configured</strong>.



<h2 id="static-properties">Static properties</h2>

You can set static properties like this (example):



```
<code class="language-php">\Stancl\Tenancy\Middleware\InitializeTenancyByDomain::$onFail = function () {
    return redirect('https://my-central-domain.com/');
};
</code>
```


A good place to put these calls is your `app/Providers/TenancyServiceProvider`'s `boot()` method.



<h3 id="tenant-model">Tenant model</h3>

`tenancy.tenant_model`



This config specifies what `Tenant` model should be used by the package. There's a high chance you're using a custom model, as instructed to by the <a href="https://tenancyforlaravel.com/docs/v3/tenants">Tenants</a> page, so be sure to change it in the config.



<h3 id="unique-id-generator">Unique ID generator</h3>

`tenancy.id_generator`



The `Stancl\Tenancy\Database\Concerns\GeneratesIds` trait, which is applied on the default `Tenant` model, will generate a unique ID (uuid by default) if no tenant id is supplied.



If you wish to use autoincrement ids instead of uuids:



<ol>
<li>set this config key to null, or create a custom tenant model that doesn't use this trait</li>
<li>update the `tenants` table migration to use an incrementing column type instead of `string`</li>
<li>update the `domains` table migration `tenant_id` column to the same type as `tenants` id</li>
</ol>

<h3 id="domain-model">Domain model</h3>

`tenancy.domain_model`



Similar to the Tenant model config. If you're using a custom model for domains, change it in this config. If you're not using domains (e.g. if you're using path or request data identification) at all, ignore this config key altogether.



<h3 id="central-domains">Central domains</h3>

`tenancy.central_domains`



The list of domains that host your <a href="https://tenancyforlaravel.com/docs/v3/the-two-applications">central app</a>. This is used by (among other things):



<ul>
<li>the `PreventAccessFromCentralDomains` middleware, to prevent access from central domains to tenant routes,</li>
<li>the `InitializeTenancyBySubdomain` middleware, to check whether the current hostname is a subdomain on one of your central domains.</li>
</ul>

<h3 id="bootstrappers">Bootstrappers</h3>

`tenancy.bootstrappers`



This config array lets you enable, disable or add your own <a href="https://tenancyforlaravel.com/docs/v3/tenancy-bootstrappers">tenancy bootstrappers</a>.



<h3 id="database">Database</h3>

<blockquote>
  If you're using Laravel Sail, please refer the <a href="https://tenancyforlaravel.com/docs/v3/integrations/sail">Laravel Sail integration guide</a>.


</blockquote>

This section is relevant to the multi-database tenancy, specifically, to the `DatabaseTenancyBootstrapper` and logic that manages tenant databases.



See this section in the config, it's documented with comments.



<h3 id="cache">Cache</h3>

`tenancy.cache.*`



This section is relevant to cache separation, specifically, to the `CacheTenancyBootstrapper`.



Note: To use the cache separation, you need to use a cache store that supports tagging, which is usually Redis.



See this section in the config, it's documented with comments.



<h3 id="filesystem">Filesystem</h3>

`tenancy.filesystem.*`



This section is relevant to storage separation, specifically, to the `FilesystemTenancyBootstrapper`.



See this section in the config, it's documented with comments.



<h3 id="redis">Redis</h3>

`tenancy.redis.*`



This section is relevant to Redis data separation, specifically, to the `RedisTenancyBootstrapper`.



Note: To use this bootstrapper, you need phpredis.



See this section in the config, it's documented with comments.



<h3 id="features">Features</h3>

`tenancy.features`



This config array lets you enable, disable or add your own <a href="https://tenancyforlaravel.com/docs/v3/optional-features">feature classes</a>.



<h3 id="migration-parameters">Migration parameters</h3>

`tenancy.migration_parameters`



This config array lets you set parameters used by default when running the `tenants:migrate` command (or when this command is executed using the `MigrateDatabase` job). Of course, all of these parameters can be overridden by passing them directly in the command call, be it in CLI or using `Artisan::call()`.



<h3 id="seeder-parameters">Seeder parameters</h3>

`tenancy.seeder_parameters`



The same as migration parameters, but for `tenants:seed` and the `SeedDatabase` job.
