                                <h1 id="quickstart-tutorial">Quickstart Tutorial</h1>

This tutorial focuses on getting you started with stancl/tenancy 3.x quickly. It implements multi-database tenancy &amp; domain identification. If you need a different implementation, then <strong>that's absolutely possible with this package</strong> and it's very easy to refactor to a different implementation.



We recommend following this tutorial just <strong>to get things working</strong> so that you can play with the package. Then if you need to, you can refactor the details of the multi-tenancy implementation (e.g. single-database tenancy, request data identification, etc).



<h2 id="installation">Installation</h2>

First, require the package using composer:



```
<code class="language-php">composer require stancl/tenancy
</code>
```


Then, run the `tenancy:install` command:



```
<code class="language-php">php artisan tenancy:install
</code>
```


This will create a few files: migrations, config file, route file and a service provider.



Let's run the migrations:



```
<code class="language-php">php artisan migrate
</code>
```


Register the service provider in `bootstrap/providers.php`:



```
<code class="language-php">return [
    App\Providers\AppServiceProvider::class,
    App\Providers\TenancyServiceProvider::class, // &lt;-- here
];
</code>
```


<h2 id="creating-a-tenant-model">Creating a tenant model</h2>

Now you need to create a Tenant model. The package comes with a default Tenant model that has many features, but it attempts to be mostly unopinionated and as such, we need to create a custom model to use domains &amp; databases. Create the file `app/Models/Tenant.php` like this:



```
<code class="language-php">&lt;?php

namespace App\Models;

use Stancl\Tenancy\Database\Models\Tenant as BaseTenant;
use Stancl\Tenancy\Contracts\TenantWithDatabase;
use Stancl\Tenancy\Database\Concerns\HasDatabase;
use Stancl\Tenancy\Database\Concerns\HasDomains;

class Tenant extends BaseTenant implements TenantWithDatabase
{
    use HasDatabase, HasDomains;
}
</code>
```


<em>Please note: if you have the models anywhere else, you should adjust the code and commands of this tutorial accordingly.</em>



Now we need to tell the package to use this custom model. Open the `config/tenancy.php` file and modify the line below:



```
<code class="language-php">'tenant_model' =&gt; \App\Models\Tenant::class,
</code>
```


<h2 id="events">Events</h2>

The defaults will work out of the box here, but a short explanation will be useful. The `TenancyServiceProvider` file in your `app/Providers` directory maps tenancy events to listeners. By default, when a tenant is created, it runs a `JobPipeline` (a smart thing that's part of this package) which makes sure that the `CreateDatabase`, `MigrateDatabase` and optionally other jobs (e.g. `SeedDatabase`) are ran sequentially.



In other words, it creates &amp; migrates the tenant's database after he's created — and it does this in the correct order, because normal event-listener mapping would execute the listeners in some stupid order that would result in things like the database being migrated before it's created, or seeded before it's migrated.



<h2 id="central-routes">Central routes</h2>

We'll make a small change to your existing route files. Specifically, we'll make sure that central routes are registered on central domains only:



```
<code class="language-php">// routes/web.php, api.php or any other central route files you have

foreach (config('tenancy.central_domains') as $domain) {
    Route::domain($domain)-&gt;group(function () {
        // your actual routes
    });
}
</code>
```


Alternatively, to keep your route files more clean, you can use <a href="https://github.com/archtechx/tenancy/pull/1180#issuecomment-2006098346">this approach</a> to register all of your routes in the `using` callback of the Application Builder.



<h2 id="central-domains">Central domains</h2>

Now we need to actually specify the central domains. A central domain is a domain that serves your "central app" content, e.g. the landing page where tenants sign up. Open the `config/tenancy.php` file and add them in:



```
<code class="language-php">'central_domains' =&gt; [
    'saas.test', // Add the ones that you use. I use this one with Laravel Valet.
],
</code>
```


If you're using Laravel Sail, no changes are needed, default values are good to go:



```
<code class="language-php">'central_domains' =&gt; [
    '127.0.0.1',
    'localhost',
],
</code>
```


<h2 id="tenant-routes">Tenant routes</h2>

Your tenant routes will look like this by default:



```
<code class="language-php">Route::middleware([
    'web',
    InitializeTenancyByDomain::class,
    PreventAccessFromCentralDomains::class,
])-&gt;group(function () {
    Route::get('/', function () {
        return 'This is your multi-tenant application. The id of the current tenant is ' . tenant('id');
    });
});
</code>
```


These routes will only be accessible on tenant (non-central) domains — the `PreventAccessFromCentralDomains` middleware enforces that.



Let's make a small change to dump all the users in the database, so that we can actually see multi-tenancy working. Open the file `routes/tenant.php` and apply the modification below:



```
<code class="language-php">Route::get('/', function () {
    dd(\App\Models\User::all());
    return 'This is your multi-tenant application. The id of the current tenant is ' . tenant('id');
});
</code>
```


<h2 id="migrations">Migrations</h2>

To have users in tenant databases, let's move the `users` table migration (the file `database/migrations/0001_01_01_000000_create_users_table.php` or similar) to `database/migrations/tenant`. This will prevent the table from being created in the central database, and it will be instead created in the tenant database when a tenant is created — thanks to our event setup. If you have any other migrations that are necessary for your application, move those migrations as well.



<h2 id="creating-tenants">Creating tenants</h2>

<blockquote>
  If you're using Laravel Sail, please refer the <a href="https://tenancyforlaravel.com/docs/v3/integrations/sail">Laravel Sail integration guide</a>:


</blockquote>

For testing purposes, we'll create a tenant in `tinker` — no need to waste time creating controllers and views for now.



```
<code class="language-php">$ php artisan tinker
&gt;&gt;&gt; $tenant1 = App\Models\Tenant::create(['id' =&gt; 'foo']);
&gt;&gt;&gt; $tenant1-&gt;domains()-&gt;create(['domain' =&gt; 'foo.localhost']);
&gt;&gt;&gt;
&gt;&gt;&gt; $tenant2 = App\Models\Tenant::create(['id' =&gt; 'bar']);
&gt;&gt;&gt; $tenant2-&gt;domains()-&gt;create(['domain' =&gt; 'bar.localhost']);
</code>
```


Now we'll create a user inside each tenant's database:



```
<code class="language-php">App\Models\Tenant::all()-&gt;runForEach(function () {
    App\Models\User::factory()-&gt;create();
});
</code>
```


<h2 id="trying-it-out">Trying it out</h2>

Now we visit `foo.localhost` in our browser, replace `localhost` with one of the values of `central_domains` in the file `config/tenancy.php` as modified previously. We should see a dump of the users table where we see some user. If we visit `bar.localhost`, we should see a different user.
